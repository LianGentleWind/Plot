\begin{abstract}
With the rapid advancement of large language models and big data technologies, the exponential growth in data volume presents significant challenges to computing architectures, particularly in the areas of data access, migration, and network communication. Traditional horizontal scaling approaches struggle to meet the increasing demands for high bandwidth and low latency. To address these challenges, this paper proposes a novel memory semantics framework based on the Unified Bus (UB) protocol, designed to facilitate efficient interconnection of heterogeneous hardware. UB architecture provides a high-bandwidth and low-lantency solution for large-scale real-time data processing, offering substantial improvements in cluster performance and resource utilization efficiency within heterogeneous environments. The memory semantics framework introduces the self-adaptive fusion of two key approaches: 1) Basic memory semantics, which enables global memory access through load/store operations and Direct Memory Access (DMA), eliminating the overhead associated with software stacks; and 2) Unified Remote Memory Access (URMA) asynchronous memory semantics, which employs a connection-less programming model to improve scalability and reduce memory usage.
\end{abstract}

\section{Introduction}
Workloads are evolving from dense models to Mixture-of-Experts (MoE) architectures, with sparse attention further increasing the fraction of fine-grained communication accesses. As a result, non-payload communication overhead is no longer negligible: latency from address generation and synchronization becomes more pronounced at finer communication granularity and higher concurrency, and highly concurrent fine-grained communication can introduce resource contention with computation, further degrading performance. Consequently, communication cost is shifting from a bandwidth-dominated regime to one jointly constrained by bandwidth and latency.

To meet the requirements of state-of-art workwords, the Unified Bus (UB) proposed by \cite{liao2025ub} can improve overall bandwidth and reduce latency and natively support memory-semantic. the UB protocol is an innovative interconnection technology that provides an efficient, flexible, and cost-effective data center network architecture \cite{zuo2025serving, liao2021ascend}. This protocol adopts a unified design, achieving efficient interconnection between heterogeneous hardware modules, covering key components such as NPU, CPU, NIC, DRAM and so on\cite{liao2021ascend}. In general, existing bus architectures follow a host-device model, where the host (CPU) manages each device and three independent memory access paths exist: host-to-device, device-to-host, and device-to-device. UB streamlines this model. It categorizes UB nodes as Users and Homes in the memory access model. A more prominent highlight is that the UB protocol introduces a connection-less reliable transmission model, enabling large-scale scalability\cite{mahitha2013deadlock}. Thus UB can provide accessing the memory resources of Homes either synchronously or asynchronously. 

The process of accessing the memory synchronously is realized by UB load/store. UB load/store eliminate the overhead of the traditional software stack through UB load/store operations, achieving global memory access in heterogeneous environments and providing underlying support for high-speed data transmission. The process of accessing the memory asynchronously is realized by URMA. URMA introduces a connection-less programming model and multi-path parallel strategy. Its minimalist connection-less memory access allows efficient communication between any nodes, significantly enhancing system scalability and reducing communication memory usage. Additionally, through out-of-order execution of unrelated memory access, URMA greatly improves concurrency performance, increasing network utilization, while adopting an active congestion control protocol that combines end and network cooperation to replace traditional passive congestion control, reducing communication latency.

However, contemporary architectures still face three critical challenges in memory semantics design. First, traditional bandwidth-intensive workloads continue to demand high-throughput communication, a requirement best served by asynchronous one-sided semantics yet unattainable through synchronous approaches. Second, emerging fine-grained communication patterns impose strict latency constraints that synchronous memory semantics can satisfy but that prove problematic for asynchronous methods. Most crucially, many real-world workloads exhibit hybrid requirements that demand simultaneous optimization of both bandwidth and latency---a fundamental challenge that neither approach can adequately address in isolation.

A natural response is to adopt integrated memory semantics. Realizing unified synchronous and asynchronous semantics requires bimodal hardware support for blocking and non-blocking primitives, a shared completion queue (CQ) that aggregates events from both modes, and unified memory address management. UB addresses these requirements by consolidating memory, messaging, maintenance, and management within a single protocol stack, thereby eliminating conversion overhead and enabling native support for fused synchronous and asynchronous semantics. Building on UB's peer-to-peer interconnect, unified instruction primitives, configurable completion notifications, and uniform programming abstraction, we propose UB-XMem, a framework for integrated synchronous/asynchronous memory semantics built upon auto-adaptive algorithms for semantic selection.

\section{Background and Motivation}
\subsection{Increasing Workload Demands on Modern Systems}
The rapid evolution of general computing and large-scale model technologies has given rise to new workload classes that impose fundamentally different demands on system memory subsystems. Modern workloads simultaneously require low latency and high bandwidth. Small-packet, low-latency communication is essential in distributed control systems\cite{eloranta2014designing,antonelli2013interconnected}, where sensor nodes and controllers continuously exchange real-time status and control messages, as well as in high-frequency trading systems that must transmit minimal-sized financial packets within microseconds to capitalize on fleeting market opportunities.

At the same time, high-bandwidth workloads are increasingly prevalent in big data processing\cite{casado2015emerging}, cloud computing, high-performance computing (HPC)\cite{reed2022reinventing}, and LLM training and inference\cite{zhou2024large}. Big data pipelines require rapid data flow between computing nodes for storage, task allocation, and aggregation. Cloud environments demand substantial bandwidth for efficient virtual machine migration and access to distributed storage resources. LLM training, as exemplified by DeepSeek's 3FS system\cite{an2024fire}, requires exceptional read-write bandwidth to sustain throughput over large AI training datasets\cite{deng2025exploring,goodman2011high}.
Current scale-out cluster architectures, however, struggle to simultaneously satisfy these stringent demands for high bandwidth and low latency. Traditional scale-out solutions have thus reached their limits, motivating scale-up expansion through memory semantics.

\subsection{Memory Semantics: Historical Evolution and Current State}

Memory semantics originally relied on CPU load/store operations over a memory bus and were subsequently enhanced by Direct Memory Access (DMA), which reduced CPU overhead. The adoption of the PCIe bus further generalized device-driven memory access through DMA engines and IOMMU, improving inter-device communication efficiency. The emergence of heterogeneous computing architectures---particularly GPU/NPU acceleration---necessitated more frequent and deeper direct memory bus access, promoting computation-data integration and boosting parallel processing performance. Latency and bandwidth limitations in existing architectures have consequently become critical bottlenecks, driving the need for advanced memory semantic solutions.

Existing memory semantics technologies fall into two broad categories: synchronous load/store approaches, typified by Compute Express Link (CXL)\cite{das2024introduction}, and asynchronous approaches, most prominently Remote Direct Memory Access (RDMA)\cite{recio2007remote}. CXL introduces synchronous memory semantics through its CXL.mem protocol, enabling standard load/store access to remote memory so that hosts can treat remote memory much like local DRAM. However, only the host can initiate transactions, and CXL 2.0 supports only a single-level switch topology, precluding cascaded multi-switch deployments.

RDMA supports asynchronous memory semantics, permitting remote nodes to perform read/write operations directly into peer memory without host CPU involvement. The Reliable Connected (RC) mode requires one Queue Pair (QP) per connection, so a process must maintain multiple QPs to communicate with multiple remote processes even within a single node. Despite enabling direct memory access, RDMA suffers from fundamental scalability limitations. In All-to-All communication patterns typical of large-scale AI training and inference\cite{rajbhandari2022deepspeed,qin2025chimera}, RDMA QP connections grow exponentially, rapidly exhausting NIC on-chip SRAM and host memory. Once SRAM is saturated, frequent PCIe accesses introduce additional latency and throughput degradation. Large-scale deployments consequently suffer from NIC resource exhaustion\cite{zhang2024high}, degraded QP performance\cite{tang2023scalable}, and fallback to less efficient protocols such as TCP\cite{shpiner2017roce,mittal2018revisiting}.
These limitations underscore the need for a new, connection-less communication architecture that delivers scalable performance without the overhead inherent in existing synchronous and asynchronous memory semantic solutions.

\subsection{The Motivation for Semantic Fusion}
As modern workloads increasingly combine divergent memory-access patterns, the central challenge is no longer optimizing a single memory semantic in isolation, but rather integrating multiple semantics within one unified system. Many applications simultaneously require low-latency load/store access for fine-grained, latency-critical operations and high-bandwidth communication for bulk data movement. Yet these two semantics remain largely isolated in current architectures, forcing workloads to trade off latency against bandwidth and preventing efficient support for phase-dependent behavior.

This observation motivates the need for integrated memory semantics. By unifying low-latency and high-bandwidth access mechanisms, systems can better match the heterogeneous demands of modern applications rather than optimizing for a single access pattern at a time. LLM inference exemplifies this need\cite{recasens2025mind,lu2025optimizing}: the prefill phase is bandwidth-intensive, whereas the decode phase is latency-sensitive, and supporting such workloads efficiently requires integration rather than the separate optimization of individual semantics.

Semantic fusion therefore provides a promising path forward. By combining the advantages of complementary memory semantics, it reduces the inefficiencies caused by semantic fragmentation and improves end-to-end performance for heterogeneous, performance-sensitive workloads.

\subsection{UB Memory Primitives}

\textbf{UB Load/Store and UBMD.}
UB introduces a globally unified \textit{UB Memory Descriptor} (UBMD), a triple \{EID, TID, UBA\} that uniquely identifies every shared resource in the bus domain.
The \textit{EID} (Entity ID) decouples the functional-entity identifier from physical location: a single physical device may host multiple entities, enabling global resource pooling, virtualization, and seamless live migration.
The \textit{TID} (Token ID) carried on every UB-Link packet simultaneously encodes the address space and the permission scope associated with the transfer.
The \textit{UBA} (UB Address) is the byte-level index of a resource within the entity's own address and permission spaces.
Whenever a data owner wishes to expose its data to remote tasks, it registers a UBMD credential in the UB system; remote accessors present this credential in every subsequent request.

\textbf{User/Home Address Translation.}
Under the \textbf{User/Home model}, the \textit{User} is the initiator of a memory access (e.g., a DMA engine or software process) and the \textit{Home} is the data owner (e.g., a host-side UB controller).
When the User issues a load/store, the local MMU first translates the user virtual address (u-VA) into a user physical address (u-PA); the \textit{UB Decoder} then maps the u-PA to a UBMD triple by computing the offset from the sender's mapping-base address.
The resulting \{EID, TID, UBA\} triple is transmitted over the bus to the Home side, where the \textit{Unified Memory Management Unit} (UMMU) resolves the UBA to a Home-side physical address and enforces access-control policies---following UB's principle of ``address resolution at the receiver.''
The UMMU can share a single page table with the host MMU, allowing a process to initiate memory accesses directly using its own virtual address; this eliminates explicit page-table creation and teardown around each transfer and removes the need for separate per-peer translation tables.
The design additionally supports lazy allocation: pages are committed on first use via a page-fault mechanism.

\textbf{URMA and Connection-less Reliable Message (RM).}
URMA's one-sided read/write operations concatenate Entity ID, Address-Space ID, and Offset into a single 64-bit address, mapping each resource to a byte-addressable URMA virtual address. Its semantics are isomorphic to load/store operations, but the transfers are asynchronous and require polling a Jetty for completion status.
Unlike RDMA's one-to-one Queue-Pair (QP) model, multiple JFS instances in the transaction layer share a single TP channel, fully decoupling network resources from host-process resources.
As a result, memory consumption for communication drops from $O(N \times P \times P)$ to $O(N)$ compared with RDMA, substantially reducing transport-related overhead.
Ordering dependencies are enforced at the transaction layer rather than the transport layer, enabling out-of-order delivery on the wire while preserving required execution-ordering guarantees.
UB further supports multipath load balancing through multiple TP channels between nodes, proactively avoiding congestion hotspots and maximizing bandwidth utilization.
The \textit{Confined Active Queue Management} (C-AQM) algorithm tightly couples the initiator and the network switch: by pre-scheduling ingress traffic via credit-based admission, C-AQM maintains near-zero buffer occupancy, enforces end-to-end flow bounding, and recycles credits instantly, achieving high throughput with near-zero queuing delay and near-linear scalability for large-scale AI training and inference.

\subsection{UB-XMem: Adaptive Semantic Fusion}

Within the architecture, UB Load/Store and URMA each occupy distinct performance sweet spots.
UB Load/Store semantics issue remote memory references directly through CPU instructions; the operations are hardware-handled, shortening the software stack and reducing cross-node latency.
Because they are built on the bus protocol, their scope is primarily limited to the bus domain; bound by CPU-instruction formats and cache-line size, Load/Store is ideal for small memory blocks and latency-critical accesses, as already demonstrated by NPUs that have adopted it for intra-device tensor parallelism.
In general-computing scenarios such as distributed file systems, directory-tree metadata---characterized by high IOPS and fine-grain access patterns---are prime candidates for Load/Store semantics.

URMA's asynchronous read/write semantics exploit UB's large memory pages and high link bandwidth, providing high reliability across multi-hop networks.
Compared to Load/Store, URMA introduces higher latency and a thicker software stack, incurring greater overhead for high-frequency, fine-grained exchanges; conversely, larger file-metadata blocks and bulk data transfers are better served by URMA's bulk-transfer path.
The optimal choice therefore depends on concrete hardware topology and workload access characteristics.

Implementing semantic selection at the application layer incurs significant overhead: applications cannot efficiently determine the appropriate semantic in real time, nor can they observe current bandwidth utilization or RTT. Realizing this mechanism entirely in hardware, on the other hand, is overly rigid and inflexible. UB-XMem therefore introduces a \textit{semantic fusion layer} that provides adaptive, efficient, and flexible semantic selection, dynamically maximizing bulk-transfer throughput, ensuring efficient bandwidth utilization, and minimizing latency across diverse workloads, QoS requirements, system constraints, and traffic patterns. The goal is to identify, for every workload, the optimal semantic sweet spot under UB.

The adaptive selection between URMA and UB Load/Store is driven by transfer size, multi-path RTT, and current bandwidth utilization. The framework first estimates the RTT of all available paths and selects the path with the lowest latency (or an optimal latency--bandwidth weighted trade-off). On the selected path, the switching threshold is then determined dynamically. Formally, we define the threshold function as \( T(\mu, RTT, ...) \), where \( \mu \) denotes current bandwidth utilization and $RTT$ denotes path delay. This function can be determined empirically or analytically by profiling the system or modeling its behavior under varying bandwidth conditions, thereby tracking the shifting crossover point at which one semantic becomes preferable to the other. When the transfer size exceeds this threshold, URMA is selected; otherwise, Load/Store is applied. By continuously adapting the threshold, the framework ensures efficient, real-time switching between URMA and Load/Store across varying workloads and network conditions, improving overall memory system performance in dynamic environments.

\section{Summary}
This paper presents a comprehensive memory semantics framework based on the UB protocol to address the pressing challenges of bandwidth, latency, and data capacity in modern computing tasks. The UB protocol facilitates efficient interconnection among heterogeneous hardware components including NPUs, CPUs, NICs, and DRAM through a unified UB design, thereby enhancing system performance in large-scale data environments. The proposed UB-XMem integrates memory and communication operations, providing adaptable support across diverse scenarios through two paradigms: load/store semantics communication and URMA semantics. It represents an advancement in memory management architectures and provides innovative solutions for large-scale data processing.

\subsection{\LaTeX-Specific Advice}

Please use ``soft'' (e.g., \verb|\eqref{Eq}|) cross references instead
of ``hard'' references (e.g., \verb|(1)|). That will make it possible
to combine sections, add equations, or change the order of figures or
citations without having to go through the file line by line.

Please don't use the \verb|{eqnarray}| equation environment. Use
\verb|{align}| or \verb|{IEEEeqnarray}| instead. The \verb|{eqnarray}|
environment leaves unsightly spaces around relation symbols.

Please note that the \verb|{subequations}| environment in {\LaTeX}
will increment the main equation counter even when there are no
equation numbers displayed. If you forget that, you might write an
article in which the equation numbers skip from (17) to (20), causing
the copy editors to wonder if you've discovered a new method of
counting.

{\BibTeX} does not work by magic. It doesn't get the bibliographic
data from thin air but from .bib files. If you use {\BibTeX} to produce a
bibliography you must send the .bib files. 

{\LaTeX} can't read your mind. If you assign the same label to a
subsubsection and a table, you might find that Table I has been cross
referenced as Table IV-B3. 

{\LaTeX} does not have precognitive abilities. If you put a
\verb|\label| command before the command that updates the counter it's
supposed to be using, the label will pick up the last counter to be
cross referenced instead. In particular, a \verb|\label| command
should not go before the caption of a figure or a table.

Do not use \verb|\nonumber| inside the \verb|{array}| environment. It
will not stop equation numbers inside \verb|{array}| (there won't be
any anyway) and it might stop a wanted equation number in the
surrounding equation.

\subsection{Some Common Mistakes}\label{SCM}
\begin{itemize}

\subsection{Identify the Headings}\label{ITH}
Headings, or heads, are organizational devices that guide the reader through 
your paper. There are two types: component heads and text heads.

Component heads identify the different components of your paper and are not 
topically subordinate to each other. Examples include Acknowledgments and 
References and, for these, the correct style to use is ``Heading 5''. Use 
``figure caption'' for your Figure captions, and ``table head'' for your 
table title. Run-in heads, such as ``Abstract'', will require you to apply a 
style (in this case, italic) in addition to the style provided by the drop 
down menu to differentiate the head from the text.

Text heads organize the topics on a relational, hierarchical basis. For 
example, the paper title is the primary text head because all subsequent 
material relates and elaborates on this one topic. If there are two or more 
sub-topics, the next level head (uppercase Roman numerals) should be used 
and, conversely, if there are not at least two sub-topics, then no subheads 
should be introduced.

with units. In the example, write ``Magnetization (A/m)'' or ``Magnetization 
\{A[m(1)]\}'', not just ``A/m''. Do not label axes with a ratio of 
quantities and units. For example, write ``Temperature (K)'', not 
``Temperature/K''.

\section*{Acknowledgment}

The preferred spelling of the word ``acknowledgment'' in America is without 
an ``e'' after the ``g''. Avoid the stilted expression ``one of us (R. B. 
G.) thanks $\ldots$''. Instead, try ``R. B. G. thanks$\ldots$''. Put sponsor 
acknowledgments in the unnumbered footnote on the first page.

\section*{References}

Please number citations consecutively within brackets \cite{b1}. The 
sentence punctuation follows the bracket \cite{b2}. Refer simply to the reference 
number, as in \cite{b3}---do not use ``Ref. \cite{b3}'' or ``reference \cite{b3}'' except at 
the beginning of a sentence: ``Reference \cite{b3} was the first $\ldots$''

Number footnotes separately in superscripts. Place the actual footnote at 
the bottom of the column in which it was cited. Do not put footnotes in the 
abstract or reference list. Use letters for table footnotes.

Unless there are six authors or more give all authors' names; do not use 
``et al.''. Papers that have not been published, even if they have been 
submitted for publication, should be cited as ``unpublished'' \cite{b4}. Papers 
that have been accepted for publication should be cited as ``in press'' \cite{b5}. 
Capitalize only the first word in a paper title, except for proper nouns and 
element symbols.

For papers published in translation journals, please give the English 
citation first, followed by the original foreign-language citation \cite{b6}.
